🗄️ 一、存储引擎 & 索引技术

📌 SIGMOD 2026
ART That Lasts: Persistent Multiversion Adaptive Radix Trees with Fast Atomic Range Queries
Mohammad Khalaji, Trevor Brown, Khuzaima Daudjee — University of Waterloo
SIGMOD '26 · 动机：持久内存（PMem）的商用化使得在持久介质上直接构建索引成为可能，但现有持久化索引要么不支持多版本（MVCC），要么无法高效支持原子范围查询——两者是事务型数据库的刚需。创新：提出持久化多版本 ART（Adaptive Radix Tree），将 MVCC 可见性信息嵌入索引节点，通过 epoch-based 内存管理实现无锁并发访问，同时支持快速原子范围扫描。为 PMem/NVM 原生数据结构提供了可直接用于生产级 OLTP 引擎的新范式。
📄 PACMMOD Vol.4 No.3
持久化索引ART
Concurrent Path-Copying Update to Tree Structures
Guanhao Hou, Dechuang Chen 等 — CUHK; HKUST
SIGMOD '26 · 动机：Path-Copying 是实现持久化树结构的经典技术（写时复制路径上的所有节点），但其在高并发多核场景下因写放大和锁争用严重退化。现有并发控制方案要么牺牲持久性语义，要么引入过高的同步开销。创新：提出基于细粒度引用计数和 epoch 回收的并发路径拷贝算法，允许并发写操作共享路径拷贝的中间结果，通过 RCU-like 机制延迟回收旧版本节点，在多核场景下实现接近无锁的吞吐量，同时保证严格持久化语义。
📄 PACMMOD Vol.4 No.3
并发索引Path-Copying
Bw-Graph: An Efficient Graph Storage System Harmonizing Topology-Aware Tree with Paged CSR
Songyao Wang, Chaokun Wang 等 — Tsinghua University
SIGMOD '26 · 动机：图存储系统长期面临"遍历 vs. 随机访问"的两难：CSR（压缩稀疏行）格式在顺序遍历（如 BFS）时性能极佳但随机访问（如点查询邻居）极差，而邻接表格式的权衡恰好相反。实际图工作负载往往两者兼有，单一格式无法兼顾。创新：提出 Bw-Graph——融合拓扑感知树（Topology-Aware Tree）与分页 CSR 的混合图存储引擎。利用图的社区结构将热点子图映射到树索引，冷数据使用压缩 CSR 分页存储，通过自适应路由在两种格式间透明切换，在遍历和随机访问两类工作负载上均取得接近最优的性能。
📄 PACMMOD Vol.4 No.3
图存储数据布局
Dynamic Flat Filter: A Unified Framework for Scalable Fingerprint-Based Filters
Haoyu Du, Yuchen Ji 等 — 苏州大学
SIGMOD '26 · 动机：LSM-Tree 和 KV 存储广泛依赖 Bloom Filter 减少不必要的磁盘 I/O，但传统 Bloom Filter 面临三个痛点：固定假阳性率无法适应动态工作负载、内存占用随数据量线性增长、删除操作需借助 Counting Bloom Filter 导致空间翻倍。创新：提出 Dynamic Flat Filter——统一指纹过滤器框架，支持动态调整假阳性率而无需重建；采用 flat 位图布局消除指针开销，空间效率超越 XOR Filter；内置可扩展的删除语义（无需 counting），特别适合 LSM-Tree 的 compaction 场景：每层可使用不同精度的过滤器，动态平衡内存与 I/O。

PartitionKV: Redesigning LSM-tree KV stores on NVMs with Adaptive Partitioning
Xingye Huang 等 — Xidian University; NWPU
SIGMOD '26 · 动机：新型非易失性内存（NVM，如 Intel Optane）拥有接近 DRAM 的延迟和持久性，理论上可消除 LSM-Tree 的写放大问题——但直接将传统 LSM 架构移植到 NVM 上会导致写停顿不减反增，因为 NVM 的读写不对称性和有限耐久度使 compaction 代价转移而非消除。创新：提出 PartitionKV——面向 NVM 的自适应分区 LSM-Tree 架构：根据数据冷热和工作负载动态将 KV 空间划分为多个独立分区，每个分区采用不同的 compaction 策略（激进/惰性/禁止）；热分区保留在 NVM 上以内存级延迟写入，冷分区下沉到 SSD；通过分区级别的写放大与读延迟联合优化模型，自适应调节分区边界。

Efficient Index Layout and Search Strategy for Large-scale High-dimensional Vector Similarity Search
Jingyu Chen 等 — 南科大; Huawei
SIGMOD '26 · 动机：百亿级高维向量相似搜索（如 1024 维 Embedding）面临两大瓶颈：第一，图索引（HNSW/NSG）的内存占用随数据量超线性增长，单机无法容纳；第二，分布式分片破坏图拓扑导致召回率下降。创新：提出两级索引布局：全局层用轻量聚类（如 IVF）做粗粒度分区，局部层在每个分区内构建高质量图索引；引入基于查询感知的路由策略，利用查询向量的分布特性只激活 2-3 个分区而非全量扫描；在十亿级 1024 维 SIFT1B/Deep1B 基准上，以 40% 的内存开销达到 99%+ 召回率（Recall@10）。

📌 FAST 2026
DMTree: Towards Efficient Tree Indexing on Disaggregated Memory via Compute-side Collaborative Design
Guoli Wei, Yongkun Li 等 — USTC
FAST '26 · 动机：分离式内存（Disaggregated Memory）架构将计算与内存物理解耦，通过 RDMA 网络访问远端内存。传统树索引（B+Tree/ART）在这种架构下遭遇灾难性性能下降——每次指针追踪都可能跨越网络，延迟从纳秒级变为微秒级。创新：提出 DMTree——计算端与内存端协同设计的树索引：在计算端缓存内部节点并预取子树，在内存端以"胖节点"形式存储连续键范围以减少网络往返；利用 RDMA one-sided read 批量拉取节点，配合计算端流水线预取隐藏网络延迟，将分离式内存下的树操作延迟降低 3-5 倍。
📄 USENIX FAST '26
分离式内存树索引
"Range as a Key" is the Key! Fast and Compact Cloud Block Store Index with RASK
Haoru Zhao, Mingkai Dong 等 — SJTU
FAST '26 · 动机：云块存储（Cloud Block Store）需要维护从逻辑块地址（LBA）到物理位置的映射索引。传统方案为每个块（如 4KB）单独建立索引项，当存储规模达到 PB 级时，索引元数据本身可能占用数百 GB 内存，成为系统瓶颈。创新：提出 RASK（Range as Key）索引：利用块存储 I/O 的空间局部性，将连续 LBA 范围映射为单个索引条目，通过自适应范围合并算法在索引粒度和查找精度间动态平衡。核心洞察是"Range as a Key is the Key"——用范围而非单个地址作为索引的基本单位，在真实云工作负载下将元数据量减少一个数量级，同时保持亚毫秒级查找延迟。

Holistic and Automated Task Scheduling for Distributed LSM-tree-based Storage
Yuanming Ren, Siyuan Sheng 等 — USTC; CUHK
FAST '26 · 动机：分布式 LSM-Tree 系统（如 HBase/Cassandra）中，每个副本独立执行 compaction 和 flush 操作。多副本间的 compaction 风暴会导致 I/O 争抢和缓存失效——一个副本的 compaction 可能使其他副本的热数据被驱逐出页缓存——但现有方案缺乏副本间的全局协调机制。创新：提出全局自动化任务调度框架：建立跨副本的 compaction 开销模型（考虑 LSM 层级、I/O 带宽、CPU 配额），通过约束求解器生成全局最优的 compaction 调度计划（错开各副本的 compaction 窗口、在副本间迁移 compaction 负载），将多副本互扰降低 60% 以上，尾延迟改善 2-3 倍。
📄 USENIX FAST '26

DOGI: Data Placement with Oracle-Guided Insights for Log-Structured Systems
(FAST 2026)
FAST '26 · 动机：日志结构存储（LSM-Tree/Log-Structured FS）中，数据放置策略直接影响垃圾回收（GC）效率和空间放大。现有策略通常基于简单的启发式规则（如按写入时间排序），缺乏对数据生命周期和访问模式的全局视角，导致冷热数据混杂、GC 效率低下。创新：提出 DOGI（Data Placement with Oracle-Guided Insights）——用"Oracle"（离线分析的历史访问 trace）训练数据热度预测模型，在线指导数据放置。核心思想是在写入路径上就将数据按预测的生命周期分组放置到不同的 segment，使 GC 时仅需清理"已冷却"的 segment，大幅降低写放大和 GC 停顿。

📌 CIDR 2026
Raster is Faster: Rethinking Ray Tracing in Database Indexing
Harish Doraiswamy, Jayant R. Haritsa — IISc Bangalore
CIDR '26 · 动机：GPU 数据库索引（如范围查询、空间连接）需要判断大量数据点是否落在查询范围内，传统方法使用光线追踪（Ray Tracing）逐点测试，在 GPU 上虽然并行但每点都需要独立的几何计算，效率受限。创新：提出用光栅化（Rasterization）替代光线追踪来做数据库索引操作——将查询范围渲染为 GPU 光栅化管线中的多边形，利用硬件光栅化器批量判定数据点的 inclusion（点在多边形内），一次性处理大量点。这是一种"用 GPU 图形管线做数据库运算"的创新思路，将数据库索引问题转化为 GPU 原生优化的图形问题。

Flexible I/O for Database Management Systems with xNVMe
Emil Houlborg, Pınar Tözün — ITU Copenhagen
CIDR '26 · 动机：现代 NVMe SSD 提供两种 I/O 路径：标准内核路径（libaio/io_uring，兼容性好但延迟较高）和 kernel-bypass 路径（SPDK，延迟极低但牺牲可移植性和内核集成）。数据库系统需要在两种路径间选择，但现有方案要求编译时或部署时绑定，无法在运行时根据工作负载动态切换。创新：提出基于 xNVMe 的灵活 I/O 框架——xNVMe 提供统一的 NVMe 硬件抽象层，封装 SPDK、libaio、io_uring 等后端，允许数据库系统在运行时透明切换 I/O 路径（联机事务用 SPDK 追求低延迟，后台 compaction 走 io_uring 利用内核缓存），无需修改应用逻辑。首次在数据库系统中实现了 I/O 路径与查询执行引擎的解耦。

🤝 二、分布式共识协议

📌 SIGMOD 2026
Scalable Leader Leases For Multi Consensus Groups in CockroachDB
Ibrahim Kettaneh, Tsvetomira Radeva, Rebecca Taft 等 — Cockroach Labs
SIGMOD '26 Industry · 动机：CockroachDB 的每个 Range（数据分片）对应一个 Raft 组，生产集群可达数十万个 Raft 组。每个组的 Leader 需要定期续约 Lease 以保证线性一致性读——数十万条 Lease 心跳消息导致控制面资源消耗（CPU/网络）占比超过 30%，且 Lease 到期后的重选举造成读延迟毛刺。创新：提出可扩展的多组 Leader Lease 机制：将数十万个 Raft 组的 Lease 管理拆分为分层心跳架构——同一节点上的 Range 共享一个 node-level lease，同一 store 上批量续约；引入预测性 Lease 延期（基于历史心跳间隔的统计模型提前延期，避免到期风暴），将 Lease 管理开销降低 80%，读延迟 P99 改善 5 倍。

LSM-Raft: Optimizing Raft for LSM-tree Store
Xiaojian Zhang 等 — Tsinghua; Timecho (Apache IoTDB)
SIGMOD '26 · 动机：Raft 共识协议与 LSM-Tree 存储引擎在同一节点上共享 I/O 和 CPU 资源，但传统方案将它们视为独立黑盒——Raft 的日志复制与 LSM-Tree 的 compaction 各自按固定节奏调度，导致周期性的 I/O 风暴和 CPU 争抢，严重影响系统吞吐和尾延迟。创新：提出 LSM-Raft——共识协议与存储引擎深度协同框架。Raft Leader 实时感知 LSM-Tree 各层的 compaction 状态和积压程度，据此动态调节日志同步节奏（在 compaction 高峰期降低复制频率、低谷期加速追赶），并通过 I/O 带宽预算的全局分配避免资源争抢。共识协议首次按存储引擎内部行为做适应性调整，将 LSM-Raft 组合系统吞吐提升 50% 以上，尾延迟降低 60%。

Epoch-based Optimistic Concurrency Control in Geo-replicated Databases
Yunhao Mao 等 — U Toronto; Keio University
SIGMOD '26 · 动机：地理复制（Geo-replicated）数据库的跨洲延迟通常在 100ms+，传统乐观并发控制（OCC）在这种场景下事务中止率极高——因为验证阶段需要跨洲确认，期间冲突窗口被极大拉长。悲观方案则需要跨洲加锁，吞吐量降到不可接受的水平。创新：提出基于 epoch 的 OCC 协议：将时间划分为固定长度的 epoch（如 100ms），同一 epoch 内的读写操作在本地执行无需跨洲协调，epoch 边界进行轻量级全局验证；利用 epoch 的确定性和可预测性，将冲突窗口从"跨洲 RTT"压缩到"epoch 边界验证延迟"，在跨洲部署中保证快照隔离的同时吞吐量提升 3-5 倍。

Reducing Tail Latency in Storage-Disaggregated Database Systems
Zhengrui Pang, Jiannan Wang — Purdue University
SIGMOD '26 · 动机：存储分离（Storage-Disaggregated）架构将计算与存储解耦，计算节点通过网络访问共享存储池。虽然弹性扩展能力显著提升，但网络延迟的抖动（特别是尾延迟 P99/P999）被放大——一次查询可能涉及数十次远端 I/O，任何一次网络抖动都会拖慢整体响应。创新：提出三维度尾延迟优化框架：(1) 请求调度——基于网络延迟预测的 hedge-request 机制，在检测到慢 I/O 时自动向其他副本发送冗余请求；(2) 智能预取——利用查询执行计划的 DAG 结构预先下发 I/O；(3) 副本选择——基于实时拥塞感知的副本负载均衡，将尾延迟 P99 降低 2-5 倍。

📌 CIDR 2026
Rosé: Flexible Replication With Strong Semantics For Partitioned Databases
Ioannis Zarkadas, Philip A. Bernstein, Asaf Cidon — Columbia; Microsoft Research
CIDR '26 · 动机：分区数据库通常对所有分区采用统一的复制策略（如全部 Raft 或全部 Paxos），但不同分区的访问模式（读写比、热点程度、跨区分布需求）差异巨大——为高冲突分区设计的强一致性策略对只读分区造成不必要的延迟开销，为低延迟分区设计的弱一致性策略无法满足金融分区的正确性要求。创新：提出 Rosé——允许每个分区独立选择复制协议（Raft/Chain-Replication/Primary-Backup/Quorum）的"one-size-fits-one"框架：通过形式化证明保证跨分区事务的正确性（可串行化）不受各分区协议异构性的影响；提供协议自动推荐器，根据分区访问特征和延迟 SLO 自动选择最优复制策略，为强语义与灵活性的统一提供了理论保证和工程实现。

Event Horizon: Asymmetric Dependencies for Fast Geo-Distributed Operations
Jonathan Arns 等 — TU Delft
CIDR '26 · 动机：跨地域分布式操作（如 Geo-Distributed Transactions）的传统模型基于对称依赖——所有参与者必须双向确认才能推进，导致协调延迟等于最慢参与者的 RTT 之和。实际上，许多操作只需要单向依赖（如日志追加只需下游确认收到，不需要下游等待上游），但现有协议无法表达和利用这种非对称性。创新：提出 Event Horizon——基于非对称依赖的分布式协调模型：形式化定义了操作的依赖图（Dependency Graph），区分"must-happen-before"（强制双向同步）和"should-happen-before"（仅需单向通知）两种依赖边；利用非对称依赖减少跨地域同步往返次数，在保证因果一致性的前提下将 Geo-Distributed 操作延迟降低 40-60%。

Aurora PostgreSQL Limitless Database
Dima Arkhangelskiy 等 20 人 — AWS

FAST '26 · 动机：大规模 AI 训练（千卡 GPU 集群、数周训练时间）中，GPU 故障不可避免——一次节点故障可能导致数小时的训练进度丢失。定期 Checkpoint 是唯一保障，但千亿参数模型的 GPU 显存状态高达数百 GB，全量 Checkpoint 的 I/O 开销导致训练停顿 5-15 分钟——在频繁 Checkpoint 下累计浪费可达训练总时间的 10-20%。创新：提出快速轻量 GPU Checkpoint/Restore 机制。利用 GPU 显存中张量的稀疏性和训练过程中的状态不变性（如优化器动量项的大部分变化微小），仅增量保存变化超过阈值的张量区域；同时在存储端设计 GPU 显存布局感知的并行写入路径，充分利用 NVMe SSD 的多队列并行能力。将 Checkpoint 时间从分钟级压缩至秒级，存储空间减少 3-5 倍，使高频 Checkpoint 成为可能。

AdaCheck: An Adaptive Checkpointing System for Efficient LLM Training
Weijie Liu 等
FAST '26 · 动机：LLM 训练中不同层的参数对故障恢复的重要性和变化速率差异巨大——Embedding 层和浅层 Transformer 变化缓慢，深层注意力投影矩阵则每个 step 都在剧变。固定频率的全量 Checkpoint 浪费大量 I/O 和存储资源在变化微小的参数上，而关键层的罕见剧烈变化又可能因 Checkpoint 间隔过长而丢失。创新：提出 AdaCheck——自适应 Checkpoint 系统。实时追踪每层参数的变化幅度（L2 范数增量），动态决定各层的 Checkpoint 频率：高频变化层（如注意力投影层）以 step 级粒度保存，低频变化层（如 Embedding）以 epoch 级粒度保存。同时利用层间冗余（如残差连接的参数相关性）进行增量编码压缩，在数万卡集群上将 Checkpoint 总 I/O 量降低 50-70%，而不影响故障恢复精度。

Fast Cloud Storage for AI Jobs via Grouped I/O API with Transparent Read/Write Optimizations
Yingyi Hao 等 — SJTU
FAST '26 · 动机：AI 训练作业的 I/O 模式与通用云存储 API 之间存在根本性不匹配：训练框架以大量小文件（千万级样本文件）并发读取，而对象存储 API 按单个 GET/PUT 操作计费和调度——每次读取一个 100KB 的图像样本都可能产生一次独立的网络往返和元数据查询。这导致 AI 作业的 I/O 吞吐远低于存储系统理论带宽，训练 GPU 因数据饥饿而频繁 stall。创新：提出分组 I/O API：将时间窗口内的一组逻辑相关 I/O 操作（如同一 mini-batch 的样本读取）聚合为一个 GroupedRequest 提交给存储系统，使存储端可以进行批量预取、合并元数据查询、按存储设备队列深度全局调度。同时在存储端实现透明的读优化（list-then-read 合并为 bulk-read）和写优化（小写合并为大顺序写），AI 框架无需任何代码修改即可获得数倍的 I/O 吞吐提升。

Supporting Our AI Overlords: Redesigning Data Systems to be Agent-First
Shu Liu, Ion Stoica, Matei Zaharia, Alvin Cheung, Natacha Crooks — UC Berkeley
CIDR '26 · 动机：AI Agent（如 Claude Code、Cursor、Devin）正在成为数据库系统的主要消费者——它们以程序化方式生成 SQL、迭代探索数据、跨多轮交互构建分析管线。但现有数据库系统的 API 和语义模型是为人类分析师设计的：假设单次查询、静态 Schema、SQL 文本交互——这些假设在 Agent 驱动的交互模式下全面失效。Agent 可能需要在一个探索会话中执行数百次查询、反复修正 Schema 假设、在不确定的数据上做出概率性决策。创新：提出 Agent-First 数据系统的 Vision：(1) 面向 Agent 的多轮会话式查询 API——支持查询上下文传递、不确定性感知的结果返回、增量结果流式推送；(2) Agent-Native 查询优化——将优化目标从"单查询延迟"改为"会话级信息获取效率"，允许优化器在探索性查询上主动降精度换速度；(3) 语义灵活的 Schema 模型——支持 Agent 在不完全了解 Schema 的情况下进行 Best-Effort 查询。这是 CIDR 2026 最具前瞻性的论文之一，为数据库领域指明了 AI 时代的重构方向。

🔮 五、趋势洞察与总结

01 · 共识协议与存储引擎深度协同

LSM-Raft 是本期最值得关注的论文之一——首次将共识协议的日志同步节奏与 LSM-Tree 的 compaction 状态动态联动。这标志着一个拐点：共识协议不再作为黑盒运行在存储引擎之上，而是需要感知存储引擎的内部状态。对 Kafka/AutoMQ 的启示：当 Kafka 的 __consumer_offsets 等内部 Topic 使用 Raft 复制时，同样面临 compaction 与 replication 的 I/O 争抢，LSM-Raft 的思路值得关注。

02 · 持久化索引结构走向工程落地

持久化多版本 ART、Concurrent Path-Copying、PartitionKV（NVM LSM-Tree）的集中出现，说明持久内存/分离式内存的索引设计已从"提出新结构"阶段进入"并发、持久化、生产级"的工程化阶段。DMTree 进一步将分离式内存作为一级架构约束，证明了在计算-存储分离场景下索引必须与架构协同设计。

03 · 工业数据库论文爆发

SIGMOD 2026 Industry Track 贡献了近年来最丰富的一批工业数据库论文：Aurora Limitless (多写)、ByteHouse (多模数仓)、TDSQL-Boundless (异构多表)、CoddSpeed (FPGA 加速)、LindormVector (向量引擎)。这些论文的共同点：都是在已有系统上叠加新的能力维度（多写、多模、向量、硬件加速），说明工业界已经度过"基于开源 fork 系统"的阶段，进入"系统架构深度融合新硬件/新负载"的时代。

04 · AI Infra 已成为存储顶会的 P0 方向

FAST 2026 有 43 篇论文，粗略统计：LLM 推理存储（KV Cache）3篇、AI 训练存储（Checkpoint/Mem/IO）3篇、ML 预处理存储 1篇——AI Infra 占比超过 16%。SIGMOD 2026 Research 有 4 篇向量搜索/ANNS 论文，Industry 也有 LindormVector 和 ByteHouse 的多模向量能力描述。CIDR 2026 更是出现了 "Agent-First 数据系统" 的 Vision 论文。AI Infra 方向的存储/数据库论文密度在过去一年至少增加了 3 倍。

05 · "One-size-fits-one" 成复制/分布式的潜在线

从 SIGMOD 的 Epoch-based OCC (Geo 场景专项优化)、Rosé 的 per-partition 复制策略，到 Event Horizon 的非对称依赖——共识/复制/事务协议正从"通用性"转向"场景定制"。过去追求"某种协议统治一切"的思路正在让位于"为一个细分场景做最好的协议"。

06 · 向量数据库从独立品类走向融合

PostgreSQL 上的向量搜索实证研究（SIGMOD）、Lindorm 多模 NoSQL 的向量引擎（Industry）、FAVOR 的过滤无关 ANNS——信号明确：向量搜索正从独立的"向量数据库"品类回归到数据库系统的原生能力。这与 Kafka 社区讨论的 "Kafka + Vector Search" 方向高度相关。

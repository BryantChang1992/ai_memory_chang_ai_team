1. 总览与核心判断

概述：本周 Kafka 社区围绕两个议题展开了高质量讨论——KIP-1279（原生集群镜像）和 KIP-1290（机架感知最小 ISR），分别解决跨集群容灾和多集群高可用的核心诉求。KIP-1279 由 Luke Chen、Federico Valeri 等 6 人联合提交，将跨集群复制直接集成到 Broker 中，从根本上颠覆 MirrorMaker 2 的部署范式；KIP-1290 由 Karl Sorensen 提出，通过在 Produce 确认路径上增加机架维度检查，为多 AZ 集群提供明确且确定性的跨机架持久性保证。

💡 核心判断
  • KIP-1290 是"低风险高回报"的改进——适合优先落地，改动小、收益明确、向后兼容。
  • KIP-1279 是架构级变革——长期价值巨大（消除 MM2 运维成本、零损耗镜像），但复杂度极高，2.7.2 移植不可行，应作为升级到最新版本的推动力。
  • 对 AutoMQ 的影响：如果 KIP-1279 落地，社区版原生具备集群镜像，AutoMQ 的差异化需要转向更高层的自动化运维与智能路由方向。

2. KIP-1279：原生集群镜像（Cluster Mirroring）

官方链接： cwiki.apache.org/confluence/display/KAFKA/KIP-1279  |  邮件列表讨论

2.1 动机：MirrorMaker 2 的五大痛点

KIP-1279 解决了一个 Kafka 运维中反复出现的痛点——跨集群数据复制。当前方案 MirrorMaker 2（基于 Connect 框架）存在以下问题：

#痛点详解
1 运维负担 MM2 作为独立 Connect Worker 运行在 Kafka Broker 之外，需要：
  • 额外的主机采购与部署
  • 独立的 Connect 配置管理
  • MM2 升级与 Broker 升级不同步，版本矩阵复杂
2 压缩低效 源集群的压缩消息在 MM2 中被解压再重压缩——这是完全冗余的 CPU 密集型操作，直接拖慢镜像吞吐并增加延迟。
3 有损 Offset 翻译 MM2 无法维护完整的源→目标 Offset 映射表（内存不足），当精确映射不可用时，返回的偏移量保证 不晚于 实际记录（即只会回退、不会跳过）。Failover 时消费端大量重复处理，高吞吐 Topic 尤其严重。
4 外部 Offset 复杂性 Flink/Spark Streaming、Kafka Connect Source、事务化应用场景中，消费端 Offset 存储在外部系统。Failover 时必须额外查询 MM2 的 __consumer_offsets 内嵌主题做 Offset 翻译。这增加了一个故障点和操作复杂度。
5 ULE 处理不当 MM2 无法优雅处理源集群的 Unclean Leader Election。源端发生 ULE 后数据丢失，目标端可能仍然保留已丢失的数据——导致两集群数据状态分歧，这是 DR 场景中不可接受的。
✅ KIP-1279 的解决思路 将跨集群复制逻辑直接集成到 Broker 内部，以原生协议进行镜像。核心价值：零外部依赖、零压缩损耗、精确 Offset 保持、元数据自动同步、快速 Failover 无 Offset 翻译。

2.2 核心架构设计

镜像角色模型

KIP-1279 引入两个新的逻辑角色，直接在 Broker 内部实现跨集群复制协议：

角色职责
Mirror Leader 源集群的 Partition Leader。向 Mirror Follower 提供数据，与普通 Follower Fetch 协议复用（基于 Leader Epoch）。
Mirror Follower 目标集群的消费者角色。像普通 Follower 一样从源集群 Mirror Leader Fetch 数据并写入本地日志，保持相同的 Offset 和 Producer ID。

🏗️ 核心设计原则

  1. 异步复制，不影响源端延迟——Producer 在源集群等待 ISR ACK（如 acks=all），数据异步 Mirror 到目标集群。源端吞吐和延迟完全不受镜像影响。
  2. Native Byte Copy——Mirror Follower 直接复制源集群的字节流写入本地日志，不解压不重压缩。这是相比 MM2 最关键的性能优势。
  3. Offset 保持——源集群和目的集群的 Offset 完全一致。Failover 后 Consumer 无需任何 Offset 翻译——这就是精确映射（而非概率性回退）。
  4. Delta Failback——目标集群的 Mirror Follower 实际上维护了与源集群的 Fetch 关系。Failback 时只需从上次 Fetch 位置继续，实现增量反向镜像。
  5. 版本兼容——源端支持 2.1+（DR/Migration 场景下，利用 4.0 引入的 Client/Broker Forward Compatibility）。目标端需要 4.1+。
🔧 新增组件一览
  • Network Protocol：新的 Mirror Fetch API（类似 Fetch API，但携带额外 Mirror 元数据）
  • Admin API / CLI：kafka-mirrors.sh --create --bootstrap-server src:9092 --to-bootstrap-server dst:9092
  • Internal Topic：__cluster_metadata（Topic 配置、Consumer Group Offsets、ACL 的同步载体）
  • Control Batch Type 7：MIRROR_PID_RESET——在镜像停止时清理 Producer State
  • Mirror State Machine：每个 Partition 的 Mirror 生命周期（STARTED → STOPPING → STOPPED）

2.3 PID 映射：状态变换 → Barrier 控制批

这是 KIP-1279 最精巧的设计之一。两个独立集群可能给不同 Producer 分配相同的 PID（64 位长整型），直接复制会导致 ProducerStateManager 混淆。

旧方案：状态性 PID 变换

transform(pid) = -(pid + 2)

# 问题：链式镜像中变换不可逆
A: pid=5   →   B: pid=-7   →   C: -((-7)+2) = 5  ❌ 碰撞!

链式镜像（A→B→C）中，PID 在 C 端恢复成原始值，与本地 Producer 产生碰撞，导致：

  • 同 Epoch 批量被静默覆盖——无声数据损坏
  • 不同 Epoch 不触发 Fencing 异常——仅一条 WARN 日志
  • 事务交织——一个 Producer 的 Commit Marker 关闭了另一个 Producer 的事务

新方案：MIRROR_PID_RESET 控制批（Type 7）

核心洞察：镜像期间目标分区只读，所有 ProducerState 条目都来自镜像数据。Failover 时所有条目都已失效——可以安全地全部过期。

# Mirror Partition 状态机
MIRRORING (read-only) → STOPPING → STOPPED (writable)

# STOPPING 阶段写入 Barrier
┌─────────────────────────────────────┐
│ MIRROR_PID_RESET 控制批 (Type: 7)    │
│ Version: 0                          │
│ SourceClusterId: "uuid-of-source"    │
└─────────────────────────────────────┘

# 效果
A: pid=5(A来源) → B: pid=5(A来源)  # 保持原始 PID
                  Barrier → 清空所有 PSM 条目
  
B writable: pid=5(B新Producer) → C: pid=5(B来源)  # 无碰撞!

控制批对 Consumer 透明（通过 isControlBatch 过滤，类似事务 Marker），可以在任何镜像拓扑下正确工作：Active-Passive、Failback、Fan-out、Fan-in、Chain。

Record Schema

{
  "type": "data",
  "name": "MirrorPidResetRecord",
  "validVersions": "0",
  "fields": [
    { "name": "Version", "type": "int16" },
    { "name": "SourceClusterId", "type": "string",
      "about": "源集群 UUID，用于数据溯源与验证" }
  ]
}

2.4 Unclean Leader Election 处理

这是 KIP-1279 的第二个深度子问题。ULE 会导致源端数据丢失，但 Mirror Follower 本地仍保留旧数据——两集群数据不可收敛。

核心机制：Last Mirrored Leader Epoch (LMLE)

LMLE 是源集群识别的"两个集群在 Leader Epoch 视角下已同步"的最新 Leader Epoch。每次反向镜像（Failback）时：

  1. 旧 Source 首先查询目标集群的 LMLE
  2. 截断本地 Log 中所有 Leader Epoch > LMLE 的记录
  3. 然后从目标集群 Fetch 数据，实现日志收敛

新增 Topic-Level 配置：mirror.support.unclean.leader.election

配置默认值行为
mirror.support.unclean.leader.election false 当 true 时，LMLE 截断会等待 All Replicas（非仅 ISR）完成截断。确保即使在之后的 ULE 中，非 ISR 副本成为 Leader 也不会导致数据不一致。
⚠️ 注意事项 启用此配置后，如果有副本因网络/磁盘问题无法追上 Leader，镜像会处于 PENDING 或 FAILED 状态。运维需手动处理或重分配副本。

2.5 元数据同步与 Failover 流程

自动同步范围

  • Topic 配置（partition count、retention、compaction 等）
  • Consumer Group Offsets——Failover 后 Consumer 直接恢复，无需 Offset 翻译
  • ACL——安全策略自动同步

Failover 流程

  1. 停止源集群的 Mirror，进入 STOPPING 状态
  2. 写入 MIRROR_PID_RESET Barrier
  3. 目标分区变为 Writable（STOPPED 状态）
  4. Consumer 直接使用已同步的 Group Offset 恢复消费
  5. （可选）源恢复后，从目标增量反向镜像（Delta Failback）

监控

所有 Mirror 指标通过标准 Kafka Broker JMX 暴露：Mirror Lag、Replication Rate、Partition State、ULE Detection 等，与现有 Replication Metrics 统一。

2.6 向 Kafka 2.7.2 移植评估

❌ 结论：不可行，不推荐尝试 KIP-1279 是一个全栈架构级变更，涉及 Network Protocol、Storage Format、KRaft Metadata、Admin API、Consumer Group Protocol 等多个层面，不能被合理 Backport 到 2.7.2。

根本原因

依赖项2.7.2 状态KIP-1279 要求
KRaft (KIP-500) Early Access（不推荐生产） 必须——Mirror Metadata 存储在 KRaft 中，ZooKeeper 元数据模型不支持集群镜像语义
Leader Epoch 协议 (KIP-101) ✅ 已支持 ✅ 已支持（但 LMLE 概念不存在）
Topic ID (KIP-516) ❌ 不存在 必须——Topic ID 是跨集群唯一标识的关键
Control Batches (KIP-360) Support Transaction Marker (Type 0-5) 需新增 Type 7 (MIRROR_PID_RESET)
Mirror Protocol Fetch API (无 Mirror 语义) 必须——需要新增 Mirror Fetch API，携带 Source Cluster ID、LMLE 等元数据
KRaft Metadata Topic ❌ ZK-based Metadata 必须——Mirror 状态、LMLE 数据需要强一致存储
Admin API / CLI AdminClient 基础 API 需新增 Mirror Admin API（`kafka-mirrors.sh`）
📋 2.7.2 用户的替代路径

如果必须在 2.7.2 上实现集群镜像，建议：

  1. 短期：优化现有 MM2 部署（增加 MM2 Worker 资源、启用压缩选项、定期重建 Offset 同步表）
  2. 中期：使用 MM2 的 emit.heartbeats.enabled=true + sync.group.offsets.enabled=true 减少 Offset 翻译的损耗
  3. 长期：规划升级到 4.1+（KIP-1279 目标版本），将集群镜像作为升级的推动力

3. KIP-1290：机架感知最小 ISR

官方链接： cwiki.apache.org/confluence/display/KAFKA/KIP-1290  |  作者：Karl Sorensen

3.1 动机：min.insync.replicas 的困境

Kafka 已支持机架感知（broker.rack，KIP-36）和机架感知 Consumer Fetch（KIP-392），但在 Produce 确认路径上仍缺少机架维度的保障。

🎯 问题场景

一个 3-AZ 集群（AZ = A, B, C），RF=5：

  • 操作者想要保证每条写入消息至少跨 2 个 AZ 持久化
  • 但目前只能设置 min.insync.replicas——这是一个数量概念，而非分布概念
  • min.insync.replicas=3 只能概率性地保证跨 AZ（有可能 3 个副本全在 AZ-A）
  • min.insync.replicas=4 可能不够（取决于分布），而且每个多余 ISR 都增加 Producer 延迟

核心矛盾：min.insync.replicas 混淆了两个正交维度：

  • 耐久性（几个副本有数据）
  • 故障域覆盖（副本跨几个 AZ）
🔎 现有替代方案对比
  • Confluent Observer Replicas：提供异步跨 DC 复制，但属于私有功能、不可在 Apache Kafka 中获取，且引入新的 Replica 类型和复制语义
  • KIP-1290：仅在 ISR 校验路径增加机架检查——最小化侵入性，不改变 Replication Protocol

3.2 核心设计与公共接口

新 Broker 配置

配置类型默认描述
min.insync.racks int 1 acks=all 时，Produce 请求被确认前必须有多少不同 rack 包含 ISR。
1=关闭（默认）；≥2 启用机架感知检查。
需要 broker.rack 在所有 Broker 上配置。

新 Error Code

Error Code描述
NOT_ENOUGH_RACKS ISR 跨的 rack 数量少于 min.insync.racks。Produce 请求被拒绝，直到更多 rack 上的副本加入 ISR。
此错误继承自 InvalidMetadataException（与 NotEnoughReplicasException 相同父类），保证现有 Client 的兼容性。

新 Metrics

MetricScope描述
UnderMinRackIsrPartition1=该分区 ISR 跨越的 rack 数 < min.insync.racks
AtMinRackIsrPartition1=该分区 ISR rack 数 == min.insync.racks
UnderMinRackIsrPartitionCountBrokerUnderMinRackIsr=1 的 Leader Partition 数量
AtMinRackIsrPartitionCountBrokerAtMinRackIsr=1 的 Leader Partition 数量

3.3 实现细节

ISR Rack 计数（无额外网络开销）

Broker 已通过 broker.rack 知晓每个副本的 Rack 信息。Rack 检查仅在 Produce ACK 路径上做一次本地 O(n) 计算：

def countDistinctRacks(isr: Set[BrokerEndpoint]): Int = {
  isr.map(_.rack).filterNot(_.isEmpty).size
}
// 复杂度: O(n), n = ISR size (通常 3-5)
// 无线程新增、无网络调用、无协议变更
// 仅当 min.insync.racks > 1 时执行

Produce 确认路径变更

// 2.7.2 现有逻辑 (ReplicaManager.checkEnoughReplicasReachOffset)
if (minIsr <= curMaximalIsr.size)
  (true, Errors.NONE)
else
  (true, Errors.NOT_ENOUGH_REPLICAS_AFTER_APPEND)

// KIP-1290 扩展后逻辑
if (minIsr <= curMaximalIsr.size) {
  if (min.insync.racks > 1) {
    val rackCount = countDistinctRacks(curMaximalIsr)
    if (rackCount >= min.insync.racks)
      (true, Errors.NONE)
    else
      (true, Errors.NOT_ENOUGH_RACKS)
  } else {
    (true, Errors.NONE)
  }
} else {
  (true, Errors.NOT_ENOUGH_REPLICAS_AFTER_APPEND)
}
💡 设计要点
  • min.insync.replicas 和 min.insync.racks 同时满足才确认——两个阈值互补
  • NotEnoughReplicasException 和 NotEnoughRacksException 两个 Error Code 分开——运维可立即区分故障类型
  • 未配置 broker.rack 的 Broker 被视为属于同一个无名 Rack（空字符串）

Rack 故障处理流程

监控告警：通过 UnderMinRackIsrPartitionCount Metric 检测到机架多样性不足
评估：操作者判断故障时长。短暂故障（分钟级）不需要动作；长时故障进入步骤 3
动态降级：kafka-configs.sh --bootstrap-server localhost:9092 --alter --topic my-topic --add-config min.insync.racks=1
恢复：Rack 恢复、ISR 扩容后，恢复原始 min.insync.racks 值

该设计遵循了 min.insync.replicas 的现有操作模型——不自动降级，运维显式调整。

验证规则（启动时 + 动态配置变更时）

  • 若 min.insync.racks > 1 但本地 Broker 未配置 broker.rack → 拒绝启动（或拒绝动态变更）
  • min.insync.racks 必须 ≥ 1
  • 若 min.insync.racks 超过集群中已知的 Rack 数量 → WARN（不拒绝——Rack 可能后续新增）

3.4 向 Kafka 2.7.2 移植评估

✅ 结论：可行，改动小，风险可控 KIP-1290 是局部修改——仅在 Produc ACK 路径上增加 Rack 计数检查。不涉及网络协议变更、不修改存储格式、不改变 ISR 协议。

具体移植路径

新增 min.insync.racks 配置

在 kafka/server/KafkaConfig.scala（2.7.2 为 Scala 代码库）中添加新配置项，默认值为 1。匹配 TopicConfigs 中的 per-topic override 路径。

// kafka/server/KafkaConfig.scala
val MinInSyncRacksProp = "min.insync.racks"
val MinInSyncRacksDoc = "When acks=all, min number of distinct racks..."

// kafka/log/LogConfig.scala  
val MinInSyncRacksProp = "min.insync.racks"
val DEFAULT_MIN_IN_SYNC_RACKS = 1
var minInSyncRacks: Int = DEFAULT_MIN_IN_SYNC_RACKS
// 添加到 LogConfig 验证和 TopicConfigs 映射
新增 NOT_ENOUGH_RACKS Error Code

在 org.apache.kafka.common.errors 包中新增 NotEnoughRacksException，继承 InvalidMetadataException。

// clients/src/main/java/org/apache/kafka/common/errors/
public class NotEnoughRacksException extends InvalidMetadataException {
    public NotEnoughRacksException(String message) {
        super(message);
    }
}

// 在 Errors.java 中注册新 Error Code:
NOT_ENOUGH_RACKS(XX, "There are not enough racks...",
    NotEnoughRacksException::new),
修改 ReplicaManager.checkEnoughReplicasReachOffset

在 kafka/cluster/Partition.scala 的 checkEnoughReplicasReachOffset 方法中增加 Rack 计数逻辑。

// 2.7.2 中此方法在 Partition.scala:826
def checkEnoughReplicasReachOffset(requiredOffset: Long): (Boolean, Errors) = {
  leaderLogIfLocal match {
    case Some(leaderLog) =>
      val curMaximalIsr = isrState.maximalIsr
      val minIsr = leaderLog.config.minInSyncReplicas
      
      // === KIP-1290 新增逻辑 ===
      val minRacks = leaderLog.config.minInSyncRacks // 从 LogConfig 获取
      // ... 现有的 minIsr 检查 ...

      if (leaderLog.highWatermark >= requiredOffset) {
        val hasEnoughReplicas = minIsr <= curMaximalIsr.size
        val hasEnoughRacks = if (minRacks > 1) {
          val distinctRacks = curMaximalIsr.flatMap { brokerId =>
            metadataCache.getAliveBroker(brokerId).flatMap(_.rack)
          }.size
          distinctRacks >= minRacks
        } else true

        if (!hasEnoughReplicas)
          (true, Errors.NOT_ENOUGH_REPLICAS_AFTER_APPEND)
        else if (!hasEnoughRacks)
          (true, Errors.NOT_ENOUGH_RACKS)
        else
          (true, Errors.NONE)
      } else
        (false, Errors.NONE)
    case None =>
      (false, Errors.NOT_LEADER_OR_FOLLOWER)
  }
}
新增 Metrics

注册 4 个新 Yammer Metrics（2.7.2 使用 Yammer Metrics 库）：UnderMinRackIsr、AtMinRackIsr、UnderMinRackIsrPartitionCount、AtMinRackIsrPartitionCount。

添加验证逻辑

在 Broker 启动和 Topic 配置变更时增加 rack 配置验证：如果 min.insync.racks > 1 但 broker.rack 未设置，log ERROR 并拒绝。

测试

参考 KIP 中的 Test Plan（多 Rack 集群、Rack 故障场景、动态降级/恢复）。

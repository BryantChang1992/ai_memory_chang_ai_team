1.6 Fluss 独有的核心能力（Kafka 2.7.2 无对应）

  • KV Store (RocksDB)：为 PK 表提供点查/更新/删除，Kafka 本质是 append-only log
  • Arrow 列式记录：支持列裁剪、谓词下推、向量化计算，Kafka 是行式字节流
  • Lake Table：数据可直接以 Parquet/Arrow 格式写入数据湖（Iceberg/Paimon）
  • Remote Log：本地仅保留 N 个 segment，老数据自动 tier 到远程存储
  • Merge Engine：行级聚合、部分更新、去重——Kafka 无此能力
  • Primary Key 表：支持 Upsert/Delete，自动生成 Changelog
  • Schema 管理：内建 schema evolution（通过 schema id）
  • 存算分离：TabletServer 可独立扩缩，CoordinatorServer 可独立高可用

    • Kafka 2.7.2 独有（Fluss 无对应）

  • Connect Framework：标准化的数据源/目标连接器生态系统
  • Streams DSL：内建流处理库（但可被 Flink/Spark 替代）
  • KRaft：KIP-500 去 ZK 化（2.7.2 中已引入 Raft 模块）
  • 事务生产者：完整的事务语义（Fluss 当前无分布式事务）
  • 幂等生产者：Producer ID + Sequence Number 机制
  • 分区的 compaction 策略：基于 key 的日志压缩

    • 2.3.4 WAL 设计（WalBuilder）

    Fluss KV 的 WAL 不是额外的文件——它直接在 changelog LogTablet 中写入！

    interface WalBuilder {
    +buildWal(walBatchData): MemoryLogRecords
    +recover(walBatchIterator): void
}

    三种实现：

    实现适用场景格式
    ----------------------
    ArrowWalBuilderLogFormat.ARROWArrow 列式 WAL
    CompactedWalBuilderLogFormat.COMPACTED压缩格式 WAL
    IndexWalBuilderLogFormat.INDEXED索引格式 WAL

    关键洞察：Fluss 的 KV 恢复不是读 WAL 文件，而是重放 changelog LogTablet。这是与 RocksDB 标准 WAL 的核心差异——它把 WAL 复用为 LogTablet 的 segment，实现 Write-Once，Read-Multiple。

    2.3.5 Snapshot 机制

    完整的 Snapshot 子系统包含近 30 个文件，核心流程：

    PeriodicSnapshotManager (定时触发)
  → KvTabletSnapshotTarget (执行快照)
  → RocksIncrementalSnapshot (RocksDB 增量快照)
  → KvSnapshotDataUploader (上传至远程存储)
  → CompletedKvSnapshotCommitter (提交快照元数据)
  → ZooKeeperCompletedSnapshotHandleStore (ZK 中记录)
  • 增量快照：基于 RocksDB 的 Checkpoint 机制，仅传输变更的 SST 文件
  • 共享文件注册：SharedKvFileRegistry 去重已存在于远程存储的文件
  • 快照下载：KvSnapshotDataDownloader 用于 Tablet 迁移/恢复
  • 快照清理：SnapshotsCleaner 删除过期快照

    3.10 关键设计总结

    Fluss Coordinator 相比 Kafka Controller 的主要改进

  • Java 实现 vs Scala：更广泛的社区基础，更易维护
  • Curator LeaderLatch：比 Kafka 手写的 ZK 竞争逻辑更健壮
  • 事件类型扁平化：30+ 事件类型 vs Kafka 的 20+，覆盖更多场景
  • 内置重平衡：Goal 驱动模型直接内置，不需要 Kafka 的 Cruise Control 外挂
  • KV Snapshot Lease：针对 Flink 集成的独特设计
  • AutoPartitionManager：自动分区创建，Kafka 依赖手动调用 API
  • Database 命名空间：TablePath(db.table) 提供逻辑隔离层
  • 分类 RPC 通道：NotifyLeaderAndIsr / StopReplica / UpdateMetadata / NotifyKvSnapshotOffset / NotifyLakeTableOffset / NotifyRemoteLogOffsets — 六条独立通道
  • Server Tag 机制：支持资源标签，用于重平衡的机架感知和隔离
  • Remote Log Manifest：原生支持远程日志（OSS/S3），Kafka 通过 Tiered Storage KIP 后期增强

    Fluss Lake 层的设计哲学

    Fluss = Streaming Storage + Lakehouse Integration

        实时写入 (ms 级)
              ↓
    ┌─────────────────────┐
    │   Fluss Tablet      │  ← 本地 Log + KV，毫秒级读写

    └────────┬────────────┘
             │ async tiering (minute-level)

    ┌─────────────────────┐

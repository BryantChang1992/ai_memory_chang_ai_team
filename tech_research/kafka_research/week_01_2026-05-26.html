Apache Kafka

v4.3.0 正式发布 · 2026-05-22
🔥 本周重磅：Apache Kafka 4.3.0 正式发布，包含 25 个 KIP、600+ commits、147 位贡献者，是 4.x 系列又一次大规模功能更新。

📦 版本发布

2026 年 5 月 22 日，Apache Kafka 社区正式发布了 Kafka 4.3.0。这是继 2 月 4.2.0 之后的又一个大版本，涵盖 Broker、Controller、Producer、Consumer、Streams、Connect 等多个组件的全面改进。

新版本发布 4.3.0 25 KIPs 147 贡献者

同期，社区还维护着 4.1.x 和 4.0.x 长期支持线：Kafka 4.1.2（3月17日）和 Kafka 4.0.2（3月16日）分别作为 bugfix 版本发布，以及 Kafka 3.9.2（2月21日）包含 KIP-1252 修复 ZooKeeper/KRaft 行为不一致问题。

🔑 关键 KIP 解读

4.3.0 的 25 个 KIP 覆盖了以下重点方向：

  • KIP-1023 · Follower Fetch from Tiered Offset
    新增 follower.fetch.last.tiered.offset.enable 配置。启用后，新加入的 follower 从分层存储的最后一个 offset 开始拉取，大幅减少 bootstrap 时的网络传输量，提升集群弹性扩缩容效率。
  • KIP-1066 · Cordon Brokers & Log Directories
    引入 cordoned.log.dirs 配置，允许管理员将特定的 log 目录标记为"隔离"状态，新分区不会分配到这些目录上。这是安全下线 broker 或磁盘的关键操作原语。
  • KIP-1196 · Group Coordinator Append Buffer
    新增 group.coordinator.append.max.buffer.size 和 share.coordinator.append.max.buffer.size 配置，控制 coordinator 最大 buffer 使用量，并配套暴露相应的 metrics，提升大规模消费者组场景的可观测性。
  • KIP-1258 · OAuth Client Assertion
    为 client_credentials grant type 增加 client assertion 认证支持，提升与身份提供商的互操作性和安全性。
  • KIP-1251 · Assignment Epochs for Consumer Groups
    改进 member epoch 校验逻辑，避免不必要的 member fencing，减少 rebalance 抖动，提升大规模消费者组的稳定性。
  • KIP-1274 · Deprecate Classic Rebalance Protocol (Phase 1)
    启动经典 rebalance 协议的废弃流程——使用旧协议启动 consumer 时将收到警告日志。社区明确方向：向新的 consumer rebalance protocol 全面迁移。

🚀 Kafka Streams & Connect 更新

  • KIP-1271 / KIP-1285：State Stores 支持 record headers 存储，并暴露到 DSL API，为需要元数据传递的流处理场景（如 trace context 传播）提供原生支持。
  • KIP-1259：新增 state.cleanup.dir.max.age.ms 配置，Streams 启动时自动清理超过指定时长的旧状态目录，降低运维负担。
  • KIP-1270：将 ProcessingExceptionHandler 扩展到 GlobalThread，GlobalKTable 消费更有保障。
  • KIP-1239：Connect RemoteClusterUtils 支持批量 offset 翻译，MirrorMaker 跨集群迁移场景大幅提速。
  • KIP-1280：MirrorMaker 迁移到 KIP-877 新版 metrics 体系，旧 metrics 将在 5.0 移除。

⚠️ 废弃公告

以下组件/配置在 4.3 中被标记为废弃，计划在 Kafka 5.0 中移除：

  • streams-scala 模块（KIP-1244）
  • group.coordinator.rebalance.protocols 配置（KIP-1237）
  • MirrorMaker 旧版 metrics 名称（KIP-1280）

AutoMQ

v1.6.5 · 2026-05-12  |  基于 Kafka 3.9.1
📌 最新动态：AutoMQ 1.6.5 于 5 月 12 日正式发布，合并 Apache Kafka 3.9.1，引入 License 支持、多 Metrics 实例、SPI 企业扩展路由等新特性。

📦 最近版本发布

版本 发布日期 Kafka 基线 关键主题
v1.6.5 2026-05-12 3.9.1 License、多 Metrics、DevKit
v1.6.4 2026-03-10 3.x ZeroZone 增强、S3 凭证隔离
v1.6.3 2026-01-29 3.x ZeroZone Dual Mapping、Coordinator 事件

🔑 1.6.5 核心更新

  • Kafka 3.9.1 合并：将上游 Apache Kafka 3.9.1 的所有修复和改进合并到 AutoMQ 分支。AutoMQ 始终保持与社区 Kafka 的基线同步，确保用户可以获得上游的 bugfix 和安全更新。
  • License 支持（#3136）：新增 License 机制，为商业化部署提供授权管理能力，标志着 AutoMQ 在企业级商业化方向上的重要一步。
  • 多 Metrics 实例（#3205）：支持多个 Metrics 实例并存，便于不同监控系统（如 Prometheus + 内部系统）并行采集指标数据。
  • SPI 企业扩展路由（#3246）：引入基于 SPI 的企业级 API 路由机制，允许在扩展层灵活定制 API 行为，提升企业级集成能力。
  • DevKit 本地开发环境（#3262, #3270, #3273）：全新的本地开发工具链 devkit（原 devbox），一键启动本地 AutoMQ 集群用于开发和测试。支持 HOST listener（#3280）以允许宿主机客户端直连。
  • Consumer Group Offset 在线重置（#3267）：支持不中断在线 consumer 的情况下重置消费组 offset，大幅简化运维流程。

🐛 重要修复

  • 修复 KVImage 内存泄漏问题（#3255）
  • 修复 S3 WAL 恢复时 503 SlowDown 导致的恢复失败（#3198/#3245）
  • 修复 truncateFullyAndStartAt 的罕见边界 bug（#3242）
  • 改进 Active Segment 管理的错误处理和日志（#3251）
  • Connect 模块：修复 KAFKA-17719，避免将未变更的 connector 配置重复写入 config topic（#3292）

🏗️ ZeroZone 持续演进

ZeroZone（零可用区流量）是 AutoMQ 的核心差异化特性，允许跨 AZ 的数据复制零流量开销。1.6.5 之前的几个版本持续对此方向投入：

  • ZeroZone 验证测试：新增零流量传输验证测试框架（#3201）
  • Dual Mapping（1.6.3）：优化 ZeroZone 的数据路由映射机制（#3149）
  • Network Transfer Optimization（1.6.2/1.6.3）：通过 WAL 网络传输优化加速 ZeroZone 重放（#3100）
  • S3 凭证隔离（1.6.4）：默认凭证隔离机制，增强多租户安全性（#3199）

Apache Fluss (Incubating)

v0.9.1-incubating · 2026-05-04  |  多语言 SDK 0.1.0 · 2026-04-06
📌 重大里程碑：Fluss 社区 5 月继续高歌猛进——v0.9.1 修复发布、多语言 SDK 正式面世、技术博客密集输出。Fluss 正从"实时分析流存储"向"实时分析与 AI 的数据基础设施"进化。

📦 版本发布：v0.9.1-incubating

5 月 4 日，Apache Fluss 发布 v0.9.1-incubating，这是 0.9 线的第一个补丁版本，完全向后兼容 0.9.0。

补丁发布 0.9.1 Bug Fixes Helm Chart 依赖升级

🐛 关键 Bug 修复

  • Paimon IOManager 泄漏（#3193）：修复 MergeTreeWriter 中的 IOManager 未释放问题，避免 /tmp 磁盘耗尽。
  • SortMergeReader 早期终止（#3137）：修复 changelog delete 场景下的读取终止问题。
  • Partial Update NULL 处理（#3157/#3090）：修复首次插入时非目标列未置 NULL、ADD COLUMN 后 partial update/delete 的 OutOfBoundsException。
  • S3 Path-Style-Access 传播（#3165）：修复 S3 path-style-access 配置从 server 到 client 的传递问题。

⛵ Helm Chart 大幅改进

  • 支持 SASL 凭证的 existingSecret 注入
  • 支持环境变量和外部 Secret 注入
  • 新增 Pod annotations/labels + PodDisruptionBudget
  • 新增调度原语：nodeSelector / tolerations / affinity
  • 重新引入 Quickstart 镜像

📦 依赖升级

  • protobuf-java: 2.5.0 → 3.25.5（大幅安全升级）
  • kafka-clients: 3.9.1 → 3.9.2
  • log4j-1.2-api: 2.17.1 → 2.25.4

🌍 多语言 SDK 正式发布（2026-04-06）

Fluss 社区发布了 Rust/Python/C++ 客户端的 0.1.0 正式版，210+ commits 打磨。核心架构创新：三方客户端共享一个 Rust 核心引擎，负责协议协商、批处理、重试和 Apache Arrow 数据交换，外层仅做轻量语言绑定。这确保了跨语言的性能一致性和功能同步。

架构创新 Rust Python C++ Arrow

📝 技术博客密集输出

Fluss 社区在 4-5 月期间技术博客产出极高，涵盖以下重要主题：

  • 《Column Pruning in Streaming Storage》（4月22日）—— 揭示 Kafka "列裁剪"实为伪裁剪（数据仍在网络传输），Fluss 通过 Arrow IPC 列存、服务端零拷贝裁剪、客户端预 shuffle 批处理实现真正的列裁剪，裁剪 90% 列时可获 10x 读吞吐提升。
  • 《Taobao Instant Commerce: Real-Time Decisions at Scale》（4月20日）—— 淘宝即时零售的实战案例：如何处理爆炸式流量下数百亿 SKU 的实时决策，Fluss 的 Delta Join、Partial Update、Streamhouse 统一架构等能力打破了她"不可能三角"。
  • 《RoaringBitmap UV Deduplication》（4月16日）—— 多维 UV 去重的技术实践，结合 Dictionary Table + 聚合引擎实现高效基数计算。
  • 《Why Fluss Chose Rust for Multi-Language SDK》（4月9日）—— 深入阐述为何选择"单一 Rust 核心 + 多语言绑定"的 SDK 架构策略。
  • 《Fluss in the Context of AI》（3月5日）—— 明确 Fluss 的新定位："streaming storage for real-time analytics and AI"，强调其在特征存储、实时上下文、向量嵌入等 AI 工作负载中的基础设施角色。

🔭 社区趋势观察

从 0.8 → 0.9 → 0.9.1 的演进路径以及近期的博客输出，Fluss 社区的战略方向愈发清晰：

  • AI 基础设施化：不再仅仅是"流存储"，而是定位为实时智能系统的数据底座。Lance 格式支持向量嵌入存储、多语言 SDK 降低 AI 工程师接入门槛。
  • Streamhouse 架构：Fluss × Iceberg / Paimon 的 Streamhouse 方案日渐成熟，淘宝 3 PB 级生产部署（40 GB/s）验证了可行性。
  • 存储计算分离深化：Aggregation Merge Engine 将聚合状态外部化到 Fluss，Flink 作业近乎无状态——这是"Zero-State Streaming"理念的落地。
  • 生态扩展：Spark 集成、Azure 支持、Helm Chart 完善——Fluss 正在构建更完整的云原生生态。

本周综述

📌 本期要点：Apache Hudi 1.2 正式发布——原生 VECTOR / BLOB / VARIANT 类型 + Lance 格式集成，将 Lakehouse 推入多模态 AI 时代；Databricks Unity Catalog 全面 GA Iceberg v3 + 跨引擎 ABAC 治理；Apache Iceberg 1.11.0 引入服务端扫描规划与表级加密；Gravitino 1.2.1 持续强化多 Catalog 联邦。

本周是 2026 年第 24 周（6 月第二周），数据平台建设方向迎来了密集的重大发布。Apache Hudi 1.2 在 6 月 7 日正式发布，这是 Lakehouse 领域首个将多模态数据存储（VECTOR、BLOB、VARIANT）作为一等公民的正式版本。同时，Databricks 宣布 Unity Catalog 上 Managed Iceberg、Iceberg v3、Foreign Iceberg 全部 GA，并推出跨引擎 ABAC（Beta）——标志着企业级的开放 Lakehouse 治理体系已经成型。

上周我们观察到 Lakehouse 正在向多模态方向转型，本周 Hudi 1.2 的发布直接将这一趋势从"提案"变为"产品"。结合 Iceberg 1.11.0 的 v3 生产就绪和 Databricks 的生态推进，Lakehouse 正在完成从 "分析平台"到"AI 原生数据平台" 的关键升级。

🪶

Apache Hudi 1.2 发布：Lakehouse 正式拥抱 AI 多模态数据

🚀 重大版本6月7日发布多模态

Apache Hudi 1.2 于 6 月 7 日正式发布，这是本周最重量级的消息。继上周我们追踪到 RFC-100 提案后，该提案仅用不到两周就完成了从设计到发布的全过程——而且交付的远不止非结构化数据存储。Hudi 1.2 直接将 Lakehouse 扩展为一个统一的多模态数据平台。

核心新特性

特性类型说明AI 场景意义
VECTOR 类型 全新 原生嵌入向量类型，内置相似性搜索 直接在 Hudi 表上做向量检索，无需外部向量数据库
BLOB 类型 全新 存储二进制对象：图片、文档、视频帧 训练数据与元数据同表存储，消除跨系统同步
VARIANT 类型 全新 原生半结构化数据存储，无需 Schema 展平 API 响应日志、事件流、ML Feature Store
Lance 文件格式集成 集成 AI 原生列式格式，对图像嵌入和向量有原生优化 AI 训练/推理的存储层统一

为什么 Hudi 1.2 是里程碑？

Hudi 1.2 解决了 AI 时代数据平台最核心的痛点——"同步困境"（Sync Dilemma）：当前 AI 应用中，同一个业务实体可能同时存在于：

  • 结构化元数据在 Lakehouse 表中
  • Embedding 向量在 Milvus/Pinecone 等向量数据库中
  • 原始文档/图片在 S3 对象存储中
  • 半结构化数据在 MongoDB/Elasticsearch 中

这种碎片化导致数据一致性维护成本极高。Hudi 1.2 的答案是：将 VECTOR、BLOB、VARIANT 全部作为原生列类型存储在同一张 Lakehouse 表中，共享同一套事务、索引、增量处理、自动表服务（compaction/clustering/cleaning）和并发控制机制。

Lance 集成：AI 列式存储的事实标准正在形成

Hudi 1.2 的 Lance 格式集成与上周我们注意到的 Fluss Lakehouse Tiering 集成 Lance 方向一致。Lance 是专为 AI/ML 设计的列式格式，对大二进制对象、向量、图像等有原生优化。Hudi 的事务层 + Lance 的存储层，构成了一种新的架构范式。

🎯 关键判断：Hudi 1.2 的意义不只是一个版本发布——它重新定义了 Lakehouse 的边界。此前业界需要"Lakehouse + 向量数据库 + 对象存储"的多系统组合，Hudi 1.2 证明了一个表就能承载这一切。这对 AI 数据平台的架构选择会产生深远影响。

Hudi 1.2 的 Flink 增强

除 AI 多模态外，Hudi 1.2 还显著增强了 Flink 集成，包括改进的 Flink MOR（Merge-on-Read）写入性能和更好的端到端一致性保证。配合 Delta 4.2.0 的 Flink connector，两大 Lakehouse 格式都在强化流处理能力。

Apache Iceberg 1.11.0：v3 Spec 生产就绪 + 表级加密

5月19日发布v3 Spec生产就绪

Apache Iceberg 1.11.0 于 5 月 19 日发布，正式将 Iceberg v3 规范推向生产就绪。虽然发布日期略早于本周窗口，但其生态影响在本周达到高峰——Databricks 和 Snowflake 都在 6 月初完成了 v3 GA。

v3 Spec 六大核心特性

特性v2 的问题v3 的解决方案性能/治理收益
二进制删除向量 Positional delete 文件随更新堆积，读时合并开销大 每个数据文件维护紧凑 bit map 带 CDC 管线的表读取速度提升一个数量级
VARIANT 类型 JSON 存为 String 列，查询时解析或展平成几百列 原生半结构化类型，引擎可推谓词到嵌套数据 事件流、API 日志、Feature Store 场景大幅简化
GEOMETRY / GEOGRAPHY 空间数据存为 WKT 字符串，依赖引擎特定扩展 一等空间类型 + 原生空间 Join 地理位置分析场景原生支持
纳秒时间戳 微秒精度不满足高频金融/IoT timestamp_ns / timestamptz_ns 交易系统、传感器网络精准时序
默认列值 加列需全量数据重写或接受 NULL ALTER TABLE ADD COLUMN 秒级完成，旧文件读时应用默认值 TB 级表的 Schema 演化零停机
行级血缘追踪 无原生支持，依赖外部审计日志 每行记录创建/修改的 Snapshot + 操作类型 合规审计、数据治理原生能力

1.11.0 引擎层重大更新

  • Spark 4.1 支持 + DSv2 迁移：MERGE INTO 支持 WITH SCHEMA EVOLUTION，无需额外 ALTER TABLE，异步 micro-batch planner 加速 Structured Streaming
  • Flink 2.1 支持 + DynamicIcebergSink（实验性）：单个 Sink 运行时路由到多个表，按需建表和 Schema 演化，甚至支持 dropUnusedColumns
  • Flink 支持 v3 新类型：nanosecond、variant、unknown 类型全支持

服务端扫描规划（Server-side Scan Planning）

这是 1.11.0 在架构层面最重要的变化。此前引擎需要自行遍历 manifest 树、拉取文件列表、过滤分区——元数据操作在引擎端完成。1.11.0 将这些工作转移到 REST Catalog 端：

  • 引擎提交 POST …/plan 请求
  • Catalog 返回优化后的 FileScanTasks
  • 小扫描即时返回，大操作返回 plan-id 轮询，超大集通过 POST …/tasks 并行获取

意义：引擎不再直接访问对象存储的元数据层，实现了引擎与存储的更彻底解耦，同时减少了小文件元数据 IO。

表级加密

Iceberg 1.11.0 引入了内置表级加密，采用封套加密（Envelope Encryption）三级密钥体系：

  • Table Master Key 在 KMS 中，不接触 Iceberg 存储
  • Key Encryption Keys (KEK) 加密后存储在表元数据中
  • Data Encryption Keys (DEK) 每文件唯一，AES-GCM 加密数据和 manifest list

支持 Google KMS 作为首批 KMS 集成，密钥自动轮转。配合安全加固趋势，表级加密将成为面向合规场景 Lakehouse 的标配能力。

File Format API 正式化

1.11.0 完成了 File Format API 的设计。此前添加新格式（如 Lance）需要多处重复引擎特定的 switch/case 代码。统一的 File Format API 使添加新存储格式变得标准化，对 Lance 等新兴格式的生态扩展意义重大。

💡 洞察：Iceberg v3 的发布意味着三大 Lakehouse 格式在数据类型丰富度上已基本对齐——Iceberg 有 VARIANT/GEOMETRY、Hudi 有 VECTOR/BLOB/VARIANT、Delta 有 Variant/Geospatial。但对 AI 场景而言，Hudi 1.2 以 VECTOR + BLOB + Lance 组合暂时领先。

Databricks Unity Catalog：Iceberg 全面 GA + 跨引擎治理

MAJORGA企业级

Databricks 宣布 Unity Catalog 上的一系列 Iceberg 能力全面进入 GA 或 Public Preview，包括 Managed Iceberg GA、Iceberg v3 GA、Foreign Iceberg GA、跨引擎 ABAC（Beta）等。与此同时，Databricks 还提出了 Iceberg v4 与 Delta 5.0 将走向统一格式的路线图展望。

GA/P预览能力一览

能力状态说明
Managed Iceberg GA 在 Unity Catalog 中直接创建、读写、优化、治理、共享 Iceberg 表，含 Predictive Optimization 和 Liquid Clustering
Iceberg v3 GA Deletion Vectors、Row Tracking、VARIANT 类型在 Managed/Foreign/UniForm 表上全支持
Foreign Iceberg GA 注册、治理、安全查询外部 Catalog（AWS Glue、Snowflake Horizon 等）中的 Iceberg 表
Credential Vending GA 外部引擎通过 UC 获取凭证，无需直接授予存储权限
跨引擎 ABAC Beta 列掩码、行过滤、Tag-based policy 对外部 Iceberg REST 客户端生效
External Sharing → Iceberg GA 通过 Delta Sharing 协议，任何 Iceberg REST 兼容客户端可读取共享数据
Iceberg 物化视图 Gated Preview CREATE MATERIALIZED VIEW … USING ICEBERG，下游消费为原生 Iceberg 表
UNIQUE 约束 Public Preview 6 月 5 日上线，Foreign Key 可引用 UNIQUE 列，RELY 启用 Join 消除和 DISTINCT 简化

五大维度定义"最可互操作的 Iceberg Catalog"

Databricks 提出的评估框架值得关注，因为它定义了企业级 Lakehouse Catalog 应该具备的能力：

  1. Open APIs + Credential Vending：任何引擎（Spark/Trino/Flink/Snowflake/DuckDB）通过 Iceberg REST API 访问，同时不暴露存储凭证
  2. Catalog Federation：一个 UC 管理所有 Iceberg 资产的元数据，已支持 AWS Glue、Snowflake Horizon、Google BigQuery、Palantir、Salesforce 等
  3. 跨引擎 ABAC：细粒度治理策略在引擎外部执行——列掩码、行过滤、标签策略对任何通过 REST API 访问的引擎生效
  4. 开放共享：通过 Delta Sharing 协议向 Iceberg 客户端共享，已支持外部 Iceberg 表共享（Public Preview）
  5. AI 驱动优化：Predictive Optimization 自动管理 compaction、clustering、统计信息收集

Roadmap：Iceberg v4 与 Delta 5.0 走向统一

Databricks 在博客中明确提到 "Iceberg v4 and Delta 5.0 will converge on a unified open format"。这是一个极其重要的信号——如果两大格式在下一大版本实现统一，意味着企业不再需要在 Iceberg 和 Delta 之间做二选一。这对整个 Lakehouse 生态将是核弹级别的变化。

UNIQUE 约束：Lakehouse 向数据库靠拢

Databricks SQL 2026 年 6 月 5 日上线了 UNIQUE 约束支持（Public Preview）。这是一个信息性（informational）约束——不强制执行，但查询优化器可以基于 RELY UNIQUE 进行 Join 消除和 DISTINCT 简化。这是 Lakehouse 从 "数据湖" 向 "数据库级可靠性" 演进的又一标志。

🏆 判断：Databricks 的这波发布不是零散的功能更新，而是一张完整的 "开放 Lakehouse 治理蓝图"。五个维度环环相扣，从 API 开放、元数据联邦、策略治理、安全共享到 AI 优化，形成了一个闭环。对于需要多引擎、多团队、多治理域的企业级 AI 数据平台来说，这套方案是目前最完整的。

Apache Gravitino 1.2.1：多 Catalog 联邦治理持续完善

补丁版本治理

Apache Gravitino 1.2.1 于 5 月 12 日发布，这是一个以稳定性、性能和正确性为重点的补丁版本。虽然不如 Hudi 1.2 那样重磅，但它在多 Catalog 联邦治理这个方向上的持续投入值得关注。

关键改进

  • Iceberg REST 合规：修正 config endpoint prefix 以符合 Iceberg REST spec，修复 table scan planning 的 SQL 生成错误
  • Authorization 性能：优化 JCasbin 策略查找，大角色权限场景下授权性能显著提升
  • Lance Catalog：清理 GravitinoLanceTableOperations 冗余代码，Lance 集成持续打磨
  • Hive 稳定性：修复 Hive MetaStore 的 HttpClient 泄漏导致的 OOM
  • Trino Connector：修复 catalog name 带 metalake 时的 TrinoException
  • Kubernetes 部署：增加 serviceAccountName 支持，Helm Chart 启动脚本统一使用 start-gravitino.sh
💡 洞察：Gravitino 作为 Apache 基金会的多 Catalog 治理项目，其价值定位与 Databricks Unity Catalog 的 Catalog Federation 能力有重叠也有互补。Gravitino 的优势是完全开源和厂商中立，而 Unity Catalog 的优势是与 Databricks 生态的深度整合。对于非 Databricks 环境的多引擎数据平台，Gravitino 仍然是最值得关注的开源选项。

趋势分析：AI 数据平台的关键转型点

趋势综合

🔷 趋势一：Lakehouse 从"分析平台"转型为"AI 原生数据平台"

这是本周最明确的信号。Hudi 1.2 的 VECTOR / BLOB / VARIANT 三种新类型 + Lance 格式集成，不是功能叠加，而是 Lakehouse 定位的根本性变化。此前 Lakehouse 解决的是"在数据湖上跑 SQL"，现在它要解决的是"在一张表里管所有 AI 数据"。

三大格式的多模态路线对比：

格式半结构化向量/嵌入二进制对象空间数据AI 列式格式
Apache Hudi 1.2 ✅ VARIANT ✅ VECTOR + 相似搜索 ✅ BLOB — ✅ Lance
Apache Iceberg v3 ✅ VARIANT — — ✅ GEOMETRY/GEOGRAPHY 🔜 File Format API
Delta Lake 4.2.0 ✅ Variant GA — — ✅ Geospatial GA —

🔷 趋势二：Catalog 成为新的竞争高地

当表格式趋于稳定，治理层（Catalog）成为下一个差异化战场。本周 Databricks 的五维度评估框架、Gravitino 1.2.1 的持续打磨、以及 Iceberg 1.11.0 的 Server-side Scan Planning（将计算推向 Catalog），都指向同一个方向：Catalog 不再只是元数据存储，而是 Lakehouse 的智能治理中心。

特别是 Databricks 提出的跨引擎 ABAC —— 治理策略在 Catalog 层执行，而不是在每个引擎中复制——这种架构对多引擎企业环境至关重要。

🔷 趋势三：Iceberg v4 + Delta 5.0 可能走向统一

Databricks 在博客中明确提到 Iceberg v4 和 Delta 5.0 将走向统一格式。如果这成为现实，将终结 "Iceberg vs Delta vs Hudi" 的格式之争。对用户来说是好消息——不用再纠结选哪个格式；对 Hudi 来说可能构成压力——需要进一步强化差异化优势（多模态 AI 数据管理正是其最有力的差异化点）。

🔷 趋势四：从"多系统组合"到"单表承载一切"

Hudi 1.2 提出的 Unified Data Foundation 愿景——结构化数据 + Embedding + 二进制对象在同一张表中——正在挑战现有的 AI 数据架构惯例。如果这一模式被市场接受，"Lakehouse + 独立向量数据库"的组合可能会被重新评估。当然，专业的向量数据库在查询延迟和索引算法上仍有优势，但对于 Less-than-real-time 的 AI 场景，内嵌向量搜索的 Lakehouse 表可能是更简单的选择。

🎯 对 CHANG_AI_TEAM 的关键启示：
  • 多模态 Lakehouse 是确定性趋势：Hudi 1.2 已证明单表多模态存储可落地，设计 AI 数据平台时应考虑未来多模态数据在同一存储层管理的架构
  • Lance 格式值得重点跟踪：Hudi、Fluss 都在集成 Lance，它可能成为 AI 列式存储的事实标准
  • Catalog 架构需要前瞻设计：跨引擎 ABAC、Catalog Federation、Server-side planning 是下一代 Catalog 的核心能力
  • 关注 Iceberg v4 + Delta 5.0 统一进展：如果实现，将重塑整个 Lakehouse 生态格局
  • 向量数据库选型需要重新评估：如果 Lakehouse 原生支持 VECTOR 类型和相似搜索，独立的向量数据库在非低延迟场景的必要性可能下降
